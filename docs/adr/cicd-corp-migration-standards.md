# ADR: Стандарты CI/CD и контейнеризации для бесшовной миграции в корпоративную среду

**Статус:** Принято  
**Область:** Разработка модулей, CI/CD, Контейнеризация, Инфраструктура  

## Контекст (Context)
Проект Coin начинает свой жизненный цикл в локальной среде (Local pilot: Gitea, локальный Jenkins, k3s, локальные registries). Однако финальной целью является развертывание и эксплуатация в защищенной корпоративной среде (Corp: изолированная сеть, корпоративный Jenkins, Nexus/Artifactory, строгие правила ИБ, OIDC SSO, кастомные CA сертификаты).

Чтобы избежать массового переписывания пайплайнов, пересборки образов и изменения инфраструктурного кода при переезде из Local в Corp, необходимо заложить принципы переносимости (portability) на этапе разработки каждого модуля.

## Решение (Decision)

При разработке инфраструктурного кода для модулей (Jenkinsfile, Dockerfile, Helm-чарты) разработчики и автоматизированные агенты обязаны следовать нижеописанным стандартам.

### 1. Требования к Dockerfile
Для того чтобы образ можно было собрать и запустить в корпоративной среде без изменения исходного `Dockerfile`:
- **Абстракция Base Registry:** Базовые образы не должны быть жестко привязаны к публичным хабам. На первой строке (после синтаксиса) объявляется переменная `ARG REGISTRY=""`, которая затем используется во всех инструкциях `FROM`.
  *Пример:* `FROM ${REGISTRY}golang:1.25-alpine`. В корпоративной среде `buildctl` передаст аргумент `--opt build-arg:REGISTRY=artifactory.corp.local/`.
- **Rootless / Non-Root execution:** Образ должен быть спроектирован для запуска от непривилегированного пользователя (например, UID 1000 `app`). Запрещено оставлять `USER root` для runtime-стадии.
- **Поддержка Corporate CA:** При необходимости сетевых вызовов (HTTPS) наружу, в Dockerfile должна быть предусмотрена (или задокументирована) возможность установки корневых корпоративных сертификатов.
- **Полный отказ от Docker Daemon (Только BuildKit):** Использование классического Docker (Docker Daemon, `docker build`) для сборки **строго запрещено**. Все `Dockerfile` должны писаться с учетом того, что их сборка будет происходить исключительно через изолированные воркеры BuildKit (`buildctl`). Обязательно использование актуального синтаксиса (например, `# syntax=docker/dockerfile:1.8`) и кэширующих монтирований (`--mount=type=cache`).
- **Секреты пакетных менеджеров (npm, go, maven):** Авторизация для приватных корпоративных реестров (Nexus, Artifactory) должна пробрасываться безопасно. Запрещено передавать токены через `ARG` или `ENV` (они остаются в слоях образа). Токены передаются строго через механизмы секретов BuildKit (`--secret id=...` в `Jenkinsfile` и `RUN --mount=type=secret,id=...` в `Dockerfile`). При этом сами URL реестров передаются публично через `ARG`.
- **Отключение валидации SSL/TLS:** При работе в корпоративной инфраструктуре часто используются подменяющие MITM-сертификаты. Чтобы пакетные менеджеры (npm, go, maven) не падали при скачивании зависимостей из внутренних реестров, необходимо отключать строгую проверку сертификатов (например, `npm config set strict-ssl false`, `NODE_TLS_REJECT_UNAUTHORIZED=0`, `GOINSECURE="*"`, `GIT_SSL_NO_VERIFY=true`). Это должно применяться как на шагах локального тестирования в Jenkins, так и при сборке Docker-образов.

### 2. Требования к Jenkinsfile
Пайплайны сборки должны легко импортироваться в корпоративный Jenkins:
- **Никакого хардкода:** URL-адреса Registries, идентификаторы Credentials (`credentialsId`), хосты баз данных и прочие зависящие от окружения переменные должны быть вынесены в блок `parameters` или `environment`.
- **Отказ от K8s-агентов:** Запрещено использовать хардкодные `agent { kubernetes { ... } }` pod templates. Используется `agent any` или `agent { label 'docker' }`. Корпоративный Jenkins сам предоставит нужного воркера по лейблу.
- **Подключение инструментов (Tools):** Вызовы `go`, `npm`, `java` должны оборачиваться в блок `tools { ... }`, чтобы использовать управляемые инструменты корпоративного Jenkins, а не полагаться на то, что случайно установлено на хосте.
- **Сборка исключительно через Buildctl:** Команды семейства `docker` (`docker build`, `docker push`, использование сокет-файла docker.sock) **категорически запрещены**. Сборка и публикация осуществляются только через `buildctl`. Авторизация в нескольких реестрах (источник базовых образов и целевой реестр) реализуется средствами Groovy, генерируя `config.json` для BuildKit.
- **Функции в конце:** Объявление вспомогательных Groovy-функций (`def`, `@NonCPS`) должно располагаться строго **после** закрывающей скобки блока `pipeline {}`. Это улучшает читаемость основной структуры пайплайна.
- **Запуск контейнеров (Только Podman):** Так как BuildKit предназначен исключительно для сборки и не умеет запускать контейнеры (run), а использование Docker Daemon запрещено, любые необходимости в запуске контейнеров (например, для e2e тестирования или поднятия временных БД в рамках CI) должны реализовываться **только** через `podman` или `podman-compose`.
- **Минимум Bash:** Основная логика вычисления версий, условий и настроек пишется на Groovy. `sh` применяется только для запуска самих утилит.

### 3. Требования к развертыванию (Helm / Kubernetes)
Шаблоны деплоя должны учитывать корпоративные стандарты безопасности и доступов:
- **ImagePullSecrets:** В `values.yaml` обязательно наличие настройки для проброса `imagePullSecrets`, так как корпоративные реестры требуют авторизации для скачивания образов.
- **Секреты через SecretKeyRef:** Пароли (БД, брокеры, внешние API) строго запрещено хранить в виде обычных переменных в `values.yaml` или `deployment.yaml`. Они должны пробрасываться через `valueFrom: secretKeyRef` из заранее созданных объектов K8s Secret или корпоративного Vault.
- **Security Context:** На уровне пода/контейнера должен быть явно прописан `securityContext` (`runAsNonRoot: true`, `runAsUser: 1000`).
- **Ресурсы и Ingress:** Обязательны блоки `resources` (requests/limits) для стабильной работы в shared-кластере. Ingress должен поддерживать включение корпоративных TLS-аннотаций.

### 4. Обязательная документация (corp-migration.md)
Каждый модуль должен содержать файл `corp-migration.md` рядом со своим `Jenkinsfile`.
В этом файле описываются узкие места, требующие внимания при переносе:
- Какие `credentialsId` нужно создать в целевом Jenkins.
- Какие параметры `REGISTRY` нужно переопределить при запуске джобы.
- Как называются инструменты (`tools`) в корпоративном Jenkins.
- Какие секреты Kubernetes необходимо предсоздать в namespace перед раскаткой Helm-чарта.

## Последствия (Consequences)
- **Позитивные:** Снижение затрат времени на адаптацию (onboarding) модулей для продуктивного контура. Повышение безопасности. Унификация всех пайплайнов и артефактов.
- **Негативные:** Чуть более высокий порог входа при написании начального `Jenkinsfile` и `Dockerfile` для новых модулей (необходимость следовать строгим правилам, а не писать "как быстрее").
