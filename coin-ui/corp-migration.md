# Руководство по миграции coin-ui в корпоративную инфраструктуру

Данный документ описывает узкие места и настройки, которые необходимо будет адаптировать при переносе CI/CD процессов и деплоя frontend-модуля `coin-ui` в защищенную корпоративную (Corp) среду.

## 1. Сборка и Jenkinsfile

### 1.1 Registries и Учетные данные
В `coin-ui/Jenkinsfile` используются дефолтные параметры для локальной среды:
- `SOURCE_REGISTRY` (default: `'registry.local'`)
- `TARGET_REGISTRY` (default: `'registry.local:5000'`)
- `SOURCE_CRED_ID` (default: `'source-registry-creds'`)
- `TARGET_CRED_ID` (default: `'target-registry-creds'`)
- `NPM_REGISTRY` (default: `'https://registry.npmjs.org/'`)
- `NPM_TOKEN_CRED_ID` (default: `'npm-auth-token'`)

**При миграции:**
Переопределите эти значения в настройках Job'ы в Jenkins для использования корпоративных реестров (Nexus / Artifactory). Например, установите `NPM_REGISTRY` в URL вашего внутреннего Nexus NPM Proxy. Убедитесь, что указанные `CRED_ID` (включая токен для NPM) заведены в Credentials корпоративного Jenkins.

### 1.2 Названия инструментов (Jenkins Tools)
В пайплайне жестко зафиксировано использование инструмента `node`:
```groovy
tools {
    nodejs 'node-22'
}
```
**При миграции:**
В корпоративном Jenkins имя этого инструмента может отличаться (например, `NodeJS-22` или `node-corp`). Проверьте *Global Tool Configuration* в Jenkins и скорректируйте имя в пайплайне при необходимости, чтобы шаг тестирования (`npm test`) мог быть успешно выполнен.

## 2. Docker-образ (Dockerfile)

### 2.1 Подмена Base Images (ARG REGISTRY)
`Dockerfile` использует `ARG REGISTRY=""` для динамической подмены источника базовых образов `node:22-alpine` и `nginx:1.27-alpine`. Никаких ручных изменений в файле не требуется — `Jenkinsfile` сам передаст корпоративный URL через аргумент `--opt build-arg:REGISTRY=...` при вызове `buildctl`.

---

## 3. Helm-чарт и Деплой (values.yaml)

### 3.1 Секреты (ImagePullSecrets)
Перед деплоем убедитесь, что в пространстве имен кластера (namespace) существует секрет типа `kubernetes.io/dockerconfigjson` с учетными данными для скачивания образов из корпоративного Registry. Этот секрет должен быть указан в `values.yaml` в блоке `imagePullSecrets`.

### 3.2 Ingress и Сертификаты
Приложение является веб-фронтендом, поэтому Ingress-настройки критически важны для продуктивной работы:
- Потребуется добавить корпоративные аннотации Ingress (например, интеграция с WAF, OIDC/Keycloak, специфичные контроллеры балансировщиков).
- Убедитесь, что настроен блок `tls` с указанием `secretName` существующего корпоративного сертификата.

---

## 4. Запуск контейнеров (Использование Podman)
В корпоративном CI строго запрещено использование классического Docker Daemon:
- Сборка образов уже настроена и выполняется через `buildctl`.
- Если в будущем потребуется локальный запуск контейнера (например, поднятие собранного Nginx для визуальной проверки или прогон UI E2E тестов с Cypress/Playwright в контейнерах), используйте **исключительно команды `podman` или `podman-compose`**. Команды `docker` в CI-среде завершатся с ошибкой из-за отсутствия `docker.sock`.
